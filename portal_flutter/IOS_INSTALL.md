## TL;DR: что скачать и что делать

| Платформа | Что «скачать» | Как поставить |
|-----------|----------------|---------------|
| **Android** | Готовый **`Portal-Flutter.apk`** из релиза GitHub (или собери: `flutter build apk --release`) | Скачай APK → открой на телефоне → разреши установку из неизвестных источников при необходимости. |
| **iOS** | **Одного файла «как APK» в открытом релизе нет.** Либо **Mac + Xcode** (вариант A ниже), либо из CI архив **`Portal-Flutter-iOS-nosign.zip`** — это **не установщик**, его всё равно нужно **открывать в Xcode / подписывать** под свой Apple ID. |

**Самый простой путь на iPhone:** Mac → клон репозитория → в `portal_flutter` выполнить `flutter create … --platforms=ios` и `python3 tool/patch_ios_info_plist.py` → `flutter pub get` → открыть **`ios/Runner.xcworkspace`** в Xcode → Signing: свой Apple ID → подключить iPhone → **Run (▶)**. Подробно — **вариант A** ниже.

Если Mac нет — установить «как с APK» **нельзя** без чужого Mac/TestFlight/магазина; попроси кого-то со сборкой или используй Android для приёма в фоне.

## Что умеет iOS-клиент

- Отправка файлов и текста на ПК (как на Android).
- Приём с ПК по TCP **:12345**, пока приложение **на экране** (активно). При приходе файла/текста показывается **локальное уведомление** (если выдал разрешение).
- Вкладка **«История»** — SQLite, как на десктопе: приём/отправка, повтор файла, копирование.
- Вкладка **«Пиры»** — кнопка **Ping** (радар) у строки: проверка `pong` с ПК.

## Почему не «как Android в фоне»

Apple **не разрешает** держать произвольный TCP-сервер в фоне так же свободно, как Android с foreground service. После сворачивания или блокировки экрана процесс чаще всего **засыпает**, и ПК не может достучаться до телефона по тому же порту.

**Практика:** для постоянного приёма с ПК удобнее **Android** с `Portal-Flutter.apk`. На iOS оставь Portal открытым на экране (можно снизить яркость), либо отправляй **с телефона на ПК**, а обратно — с ПК, когда приложение снова открыто.

Полноценный «всегда в фоне» на iOS без обходов (VoIP/audio и т.д., с риском отклонения в App Store) в этот проект **не внедрялся**.

## Вариант A: сборка у себя в Xcode (бесплатный Apple ID)

1. На Mac: `git clone` репозитория, `cd portal_flutter`.
2. Один раз:  
   `flutter create . --project-name portal_flutter --org org.portal --platforms=ios`  
   затем:  
   `python3 tool/patch_ios_info_plist.py`
3. `flutter pub get`
4. Открой `ios/Runner.xcworkspace` в Xcode.
5. **Signing & Capabilities** → Team: свой Apple ID → для Runner выбери **Personal Team**.
6. Подключи iPhone по кабелю, выбери устройство, **Run** (▶).

На телефоне: **Настройки → Основные → VPN и управление устройством** — доверь разработчика, если система попросит.

Срок подписи **free account** — около **7 дней**, потом пересборка/переустановка.

## Вариант B: артефакт из GitHub Actions (без подписи)

В workflow **Portal Flutter Build** job **ios-nosign** собирает **Portal-Flutter-iOS-nosign.zip** (`Runner.app`).

Это **не** готовый IPA для магазина: zip нужно **подписать у себя** (Xcode / `codesign`) или собрать IPA в Xcode по шагам из варианта A. Проще клонировать репо и собрать как в A.

---

## Вариант C: TestFlight / App Store

Нужны **платный** Apple Developer Program (~$99/год), сертификаты распространения, **App Store Connect**, загрузка сборки через Xcode или Transporter. Это стандартный путь для «как у всех приложений»; автоматизацию под ваш аккаунт в репозитории не настраивали.

Кратко: в Xcode **Product → Archive** → **Distribute App** → App Store Connect → TestFlight.

---

## Сеть и пароль

- ПК и iPhone в одной сети / **Tailscale**, на ПК запущен Portal, порт **12345**, тот же **пароль сети**, что в настройках iOS-приложения.
- На iPhone в **Пиры** укажи Tailscale-IP ПК (часто `100.x.x.x`).
